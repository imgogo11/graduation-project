# 数据库 + 算法模块协同工作的系统设计说明

## 一、设计目标

本系统面向股票数据与电商商品数据的管理和分析场景，采用“**数据库存储 + 算法模块增强**”的协同设计方案。

该设计的目标如下：

1. 使用数据库完成业务数据的持久化存储与通用查询。
2. 使用算法模块完成高频区间查询、历史版本查询等特定分析任务。
3. 保证系统同时具备工程完整性、算法特色和良好的可扩展性。
4. 体现数据库技术与数据结构算法在系统中的合理分工。

---

## 二、设计思路

数据库与算法模块并不是替代关系，而是协作关系。

### 2.1 数据库负责的内容
数据库主要负责：

- 原始数据的持久化存储
- 基础增删改查
- 条件筛选与分页查询
- 用户信息与权限管理
- 日志记录与结果存档

数据库适合处理通用业务操作，例如：

- 查询某支股票的基础信息
- 查询某类商品列表
- 按条件筛选某时间段的数据
- 分页展示历史记录

### 2.2 算法模块负责的内容
算法模块主要负责：

- 高频区间统计
- 历史版本查询
- 快速最值、求和、排名类查询
- 特定分析任务的性能优化

算法模块适合处理如下场景：

- 查询某股票某时间区间的最高价、最低价、成交量总和
- 查询某商品历史价格序列的区间最值
- 查询某历史版本下的第 k 小值
- 比较两个历史时间点的数据分布差异

## 三、为什么不能只依赖数据库

关系型数据库可以完成大量常见查询，但对于某些特定场景，并不是最合适的方案。

### 3.1 数据库的优势
- 支持结构化存储
- 支持 SQL 条件查询
- 支持索引、事务和并发控制
- 易于维护业务数据一致性

### 3.2 数据库的局限
对于以下任务，若完全依赖数据库，可能存在不足：

- 高频区间最值查询
- 高频区间求和查询
- 历史版本快照查询
- 多次重复的分析型查询
- 需要明确体现算法设计能力的场景

### 3.3 引入算法模块的意义
引入线段树、主席树、分块等数据结构后，可以：

- 降低特定查询的时间复杂度
- 提高高频分析任务的响应效率
- 构建适合论文展示的算法优化模块
- 体现系统的研究性与技术特色

## 四、总体协同架构

系统整体采用如下协同结构：

```text
前端页面
   ↓
后端业务服务（Python / FastAPI）
   ↓
┌──────────────────────┬──────────────────────┐
│      数据库层         │      算法模块层       │
│      MySQL           │      C++             │
│                      │                      │
│ 原始数据存储         │ 线段树 / 主席树 / 分块 │
│ 通用条件查询         │ 区间统计 
```

该架构中：

- 前端只调用统一接口
- 后端负责判断请求应交由数据库还是算法模块处理
- 数据库存储原始数据
- 算法模块基于数据库中的数据构建分析索引

---

## 五、数据流转过程

### 5.1 数据写入流程

当系统导入或新增数据时，整体流程如下：

- 前端上传数据文件或提交录入信息
- 后端完成数据校验与清洗
- 后端将原始数据写入数据库
- 后端触发算法索引构建或更新
- 更新后的分析结构供后续查询使用

### 5.2 数据查询流程

系统查询分为两类：

#### （1）普通业务查询

对于列表展示、条件筛选、详情查看等需求：

- 后端直接查询数据库
- 返回结果给前端展示

#### （2）分析型查询

对于区间最值、区间求和、历史版本比较等需求：

- 后端先定位查询对象与查询区间
- 将查询请求交给算法模块
- 算法模块返回计算结果
- 后端封装结果并返回前端

## 六、数据库与算法模块的分工

### 6.1 数据库层职责

数据库层作为系统的基础数据中心，负责以下内容：

#### 基础数据存储
- 股票基础信息
- 股票历史价格数据
- 商品基础信息
- 商品价格历史记录
- 用户信息
- 查询日志

#### 通用业务支持
- 条件筛选
- 分页查询
- 多表关联
- 数据更新与删除
- 报表结果存档

#### 数据库层特点
- 数据可靠
- 支持持久化
- 适合通用查询
- 适合业务管理

### 6.2 算法模块层职责

算法模块层作为分析加速层，负责以下内容：

#### 分析索引构建
- 从数据库读取指定对象的时间序列数据
- 将数据转换为数组或离散化结构
- 构建线段树、主席树、分块等索引

#### 高性能分析查询
- 区间最大值、最小值查询
- 区间总和查询
- 历史版本查询
- 排名类查询
- 分布统计类查询

#### 算法模块特点
- 针对性强
- 查询效率高
- 适合高频分析任务
- 易于体现算法设计能力

## 七、典型数据结构与使用场景

### 7.1 线段树

适用于处理时间序列上的区间查询。

#### 适用场景
- 股票价格区间最大值
- 股票价格区间最小值
- 成交量区间求和
- 商品价格区间波动分析

#### 作用

将数据库中的历史序列抽取出来后，在内存中建立线段树，后续高频查询直接调用该结构完成计算。

### 7.2 主席树（可持久化线段树）

适用于处理历史版本查询。

#### 适用场景
- 查询某个历史时刻的数据分布
- 查询某历史版本中的第 k 小值
- 对比两个时间点之间的数据变化
- 支持按版本进行快照查询

#### 作用

针对按时间演化的数据构建多个版本根节点，实现版本化分析。

### 7.3 分块

适用于频繁统计、局部更新场景。

#### 适用场景
- 某价格区间商品数量统计
- 某销量区间统计
- 高频条件下的快速统计计算

#### 作用

将数据分组维护，降低多次查询与局部修改的代价。

## 八、如何基于数据库中的数据构建算法结构

算法结构不是直接建立在数据库表本身之上，而是建立在“从数据库读取出来的分析数据”之上。

### 8.1 基本过程
- 从数据库中读取目标数据
- 按查询需求进行排序、清洗、离散化
- 转换为适合算法处理的数组或版本序列
- 在算法模块中构建对应数据结构
- 保存结构引用，供后续查询调用

### 8.2 示例：股票区间查询

例如，数据库中有某支股票的历史价格表：

```text
trade_date   close_price
2024-01-01   10.5
2024-01-02   10.8
2024-01-03   10.3
...
```

后端读取后形成价格数组：

```text
[10.5, 10.8, 10.3, ...]
```

随后在算法模块中基于该数组建立线段树，用于支持：

- 区间最大值查询
- 区间最小值查询
- 区间和查询

### 8.3 示例：历史版本查询

对于商品价格历史数据，可按时间顺序逐步构建主席树版本：

- 第 1 天形成版本 1
- 第 2 天形成版本 2
- 第 3 天形成版本 3

用户查询某一时刻的数据状态时，直接访问对应版本。

## 九、算法索引的维护方式

为了保证数据库数据与算法结构的一致性，系统可采用以下几种维护策略。

### 9.1 启动时统一构建

系统启动时从数据库中读取分析数据，统一构建算法索引。

#### 优点
- 实现简单
- 结构清晰
- 适合毕业设计演示

#### 缺点
- 数据更新后需要重新构建或局部更新

### 9.2 数据写入后同步更新

每次有新数据写入数据库后，立即更新相应算法结构。

#### 优点
- 数据一致性较好
- 查询结果更实时

#### 缺点
- 实现复杂度较高

### 9.3 定时批量重建

系统定期从数据库同步数据，并重新构建分析索引。

#### 优点
- 适合分析型系统
- 易于控制更新流程

#### 缺点
- 实时性略弱

## 十、推荐实现方案

对于本项目，推荐采用以下实现方式：

### 10.1 存储层

使用 MySQL 存储所有原始业务数据。

### 10.2 业务层

使用 Python FastAPI 实现：

- 数据导入
- 数据清洗
- 数据库访问
- 查询路由分发
- 调用算法模块
- 返回前端结果

### 10.3 算法层

使用 C++ 实现：

- 线段树
- 主席树
- 分块结构
- 相关分析接口

### 10.4 协同方式

采用“数据库持久化 + 算法索引增强”的双层模式：

- 普通查询走数据库
- 分析查询走算法模块
- 后端负责统一调度

## 十一、接口调用逻辑建议

### 11.1 普通查询接口

示例：

```text
GET /api/stocks/list
GET /api/products/list
GET /api/stocks/detail/{code}
```

特点：

- 直接访问数据库
- 用于分页、筛选、详情展示

### 11.2 分析查询接口

示例：

```text
GET /api/stocks/range-max
GET /api/stocks/range-sum
GET /api/products/version-kth
GET /api/products/version-compare
```

特点：

- 调用算法模块
- 用于区间统计、版本分析等高级功能
