# 毕业设计最终任务规划（整合版）

## 1. 项目最终定位

### 1.1 推荐最终题目
**基于 C++ / Python 的多源数据管理与智能分析系统设计与实现**

可选题目：

- 面向股票与电商数据的高性能管理分析系统设计与实现
- 多源时序数据管理与算法增强分析平台设计与实现

### 1.2 项目名称（建议）
**多源数据管理与算法分析平台（StockMall-DataLab）**

### 1.3 项目定位
本项目不是一个普通的后台管理系统，而是一个面向**股票数据**与**购物平台数据**的全栈数据管理与智能分析平台。系统既要完成数据导入、清洗、存储、查询、可视化等工程任务，又要将你擅长的高级数据结构真正落地到业务分析场景中。

### 1.4 最终方案原则

- 以 `tasks-gpt5.3-codex.md` 的结构完整性和可落地性为主
- 吸收另外三份文档中关于“高性能计算引擎”“跨语言协同”“性能对比实验”的亮点
- 不建议从零实现数据库内核，而采用“数据库持久化 + Python 业务网关 + C++ 算法引擎”的协同架构
- 保证题目有足够工作量，但难度仍然控制在毕业设计可以按期完成的范围内

## 4. 总体架构设计

### 4.1 架构思路
系统采用“前后端分离 + 数据库持久化 + 算法引擎增强”的协同架构。数据库负责原始数据存储与通用业务查询，算法模块负责高频区间查询、历史版本查询和复杂离线统计。

### 4.2 总体架构图
```text
前端展示层（Vue3 + TypeScript + ECharts）
                |
                v
后端业务网关（Python / FastAPI）
        |                       |
        v                       v
数据库层（MySQL）         算法引擎层（C++）
原始数据 / 用户 / 日志     线段树 / 主席树 / 分块 / 分治
        |                       |
        +-----------+-----------+
                    |
                    v
            统一结果封装与返回
```

### 4.3 模块划分

#### 4.3.1 数据采集与导入模块（Python）

- 股票数据采集：可使用 `AkShare`、`Tushare` 或离线 CSV 数据集
- 电商数据来源：Kaggle 公开数据集、商品评论数据集、模拟订单生成器
- 支持 CSV / Excel / JSON 批量导入

#### 4.3.2 数据清洗与预处理模块（Python）

- 缺失值处理
- 异常值标记
- 时间格式统一
- 字段标准化
- 构建算法所需的数组、离散化序列和版本序列

#### 4.3.3 业务 API 模块（Python / FastAPI）

- 用户管理、权限管理、查询接口、统计接口
- 调度数据库查询或算法模块调用
- 返回统一 JSON 结果

#### 4.3.4 算法引擎模块（C++）

- 实现线段树、主席树、分块、分治算法
- 提供统一接口供 Python 调用
- 负责高频区间查询、版本查询、离线统计

#### 4.3.5 前端可视化模块（Vue3）

- 数据列表管理页面
- 股票趋势图、成交量图、品类销售图
- 条件筛选和分析结果展示
- 算法实验台与性能对比看板

#### 4.3.6 日志与运维模块

- 请求日志
- 算法调用日志
- 任务执行日志
- Benchmark 结果存档

### 4.4 数据库与算法模块协同方式

- 数据库存储原始业务数据、用户数据、日志数据和报表结果
- 算法模块不直接以数据库表为运行对象，而是基于从数据库读取出的分析数据构建内存索引
- 普通查询优先走数据库
- 高频统计、历史版本、区间排名、多维离线统计走算法模块
- 后端负责统一调度、参数校验和结果封装

### 4.5 C++ 与 Python 集成方案

#### 推荐主方案：`pybind11`

- 优点：开发链路短、调用快、部署简单、适合毕业设计按期完成
- 适合：将 C++ 算法模块直接封装为 Python 扩展，在 FastAPI 中直接调用

#### 冲刺增强方案：`gRPC`

- 优点：工程感更强，服务边界更清晰，适合答辩展示“跨语言微服务”
- 缺点：通信、部署和调试复杂度更高

最终建议：**先用 `pybind11` 跑通全链路，再把 `gRPC` 作为冲刺亮点**。

## 5. 技术选型与语言分工

### 5.1 技术栈建议

- 前端：Vue3 + TypeScript + Pinia + Element Plus + ECharts
- 后端：FastAPI + SQLAlchemy + Pydantic
- 数据处理：Pandas + NumPy
- 调度任务：APScheduler 或 Celery
- 算法层：C++17/20 + pybind11（推荐）/ gRPC（可选）
- 数据库：MySQL（推荐）+ Redis（可选）
- 部署：Docker Compose

### 5.2 为什么推荐 MySQL

- 学校环境中常见，资料多，部署方便
- 更适合毕业设计快速落地
- 若你后续想强化时序查询与 JSON 能力，可替换为 PostgreSQL

### 5.3 语言分工

- Python：业务接口、鉴权、数据采集、清洗、调度、报表导出、Benchmark 管理
- C++：高性能算法核心、复杂查询计算、历史版本索引、离线多维统计
- TypeScript：前端交互、图表展示、实验看板、系统状态展示

## 7. 核心算法落地设计

目标：让算法真正服务于系统业务，而不是停留在“刷题式展示”。

### 7.1 线段树（Segment Tree）

#### 业务场景

- 股票 K 线区间最高价、最低价、成交量总和
- 商品历史价格区间最值与均值
- 某段时间内销售额区间统计

#### 接口示例

- `range_max(entity_id, l, r)`
- `range_min(entity_id, l, r)`
- `range_sum(entity_id, l, r)`

#### 扩展点

- 支持懒标记，实现区间更新或回测模拟
- 可与朴素遍历和 SQL 聚合做性能对照

### 7.2 主席树（Persistent Segment Tree）

#### 业务场景

- 查询某历史时刻的数据分布
- 查询某时间版本下价格或销量第 k 大 / 第 k 小
- 做历史快照对比

#### 接口示例

- `query_kth(version_l, version_r, k)`
- `query_rank(version, value)`

#### 价值

- 支持版本化分析，适合论文中“历史版本查询”的核心章节
- 能明显体现 ACM/ICPC 数据结构能力

### 7.3 分块（Square Root Decomposition）

#### 业务场景

- 高频行为评分统计
- 局部更新较多但结构不适合线段树懒标记的查询
- 商品价格区间数量统计、销量分段统计

#### 接口示例

- `block_update(l, r, delta)`
- `block_query(l, r)`

#### 价值

- 体现不同数据结构的工程取舍
- 能与线段树形成复杂度和实现难度上的对比

### 7.4 分治（CDQ / 整体二分 / 归并分治）

#### 业务场景

- 多因子选股：例如 `PE < A` 且 `Turnover > B` 且 `Yield > C`
- 电商多维筛选：价格、销量、评分等维度联合过滤
- 离线批量报表分析
- 阈值类批量查询、逆序对统计、支配关系计数

#### 配套结构

- CDQ 分治 + 树状数组（Fenwick Tree）
- 整体二分 + 离散化

#### 价值

- 形成论文中理论亮点最强的一部分
- 可以直接展示复杂度优化过程

如果你要做一份**最稳妥、最容易拿高分、又最能体现算法能力**的毕业设计，那么最终建议是：

- 以“算法实验台 + 性能对比实验”作为论文与答辩的核心加分项
