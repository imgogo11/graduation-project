### 第 1 章 绪论

#### 1.1 研究背景与意义

随着证券市场数字化程度不断提高，股票历史交易数据在采集频率、时间跨度和标的覆盖范围上持续扩张。对于以日线数据为核心的教学研究和工程实践而言，仅依赖电子表格或零散脚本，已经难以同时满足数据导入、批次管理、用户隔离、统计分析和异常识别等多方面需求。互联网金融风控系统、金融风险预警系统以及证券分析软件的相关研究表明，金融数据管理与风险分析必须在统一业务流程中组织，才能形成稳定可复用的系统能力。[5][6][10]

另一方面，股票交易数据分析既具有典型的 Web 业务属性，也包含明显的时序计算特征。一方面，系统需要支持注册登录、文件上传、批次管理、记录查询和结果展示等常规工程能力；另一方面，在区间最值查询、顺序统计、联合异常排序和事件上下文解释等环节，又存在较强的性能敏感性。时序数据库关键技术综述以及面向异常检测的时序数据库查询优化研究说明，时序数据系统的查询组织方式会直接影响分析效率和结果可用性。[8][11]

跨语言协同为这一问题提供了现实路径。相关研究表明，pybind11 可以以较轻量方式建立 C++ 与 Python 之间的接口桥梁，使底层高性能模块能够被 Python 业务系统直接加载和调用。[1] 同时，t-digest、可持久化数据结构以及面向大时间序列的索引研究又分别从近似顺序统计、精确顺序统计和时序聚合查询三个角度，为性能增强提供了可落地的方法基础。[2][3][4][12]

此外，金融风险识别已经不再满足于“算出一个异常分数”，而更强调多因子联合表达、可视化呈现和上下文解释。时间序列异常检测综述、异常可视分析综述、数字化转型背景下的金融风险监测研究以及股票市场操纵预警研究都指出，异常结果若缺少比较、分级与解释，很难真正服务于风险认知与决策。[7][9][13][14] 基于此，本文围绕“交易数据管理 + 分析中心 + C++ 算法增强 + 风险雷达”设计并实现一个可运行、可验证、可扩展的股票交易数据管理与分析系统。

#### 1.2 国内外研究现状

##### 1.2.1 国内研究方向及进展

国内相关研究首先集中在金融风控平台和证券分析软件的设计与实现上。互联网金融风控系统的设计与实现、金融风险预警系统设计及实现以及时间序列与证券数据分析软件的设计与实现等工作，普遍重视业务流程整合、指标体系设计、结果展示和系统可用性，说明国内在平台型金融应用方面已经积累了较为丰富的工程经验。[5][6][10] 这类研究为本文提供了重要启示，即股票交易数据系统不能只停留在单点算法或单一页面上，而应当围绕数据接入、分析展示和风险表达构建完整闭环。

国内研究的第二个方向是围绕时序数据管理与查询优化展开。面向异常检测的时序数据库查询优化研究从异常查询场景出发，讨论了多数据源条件下的执行延迟与系统计算资源利用问题；时序数据库关键技术综述则更系统地总结了时序数据模型、压缩存储、索引和查询执行等关键环节。[8][11] 这些成果表明，在股票交易数据场景中，数据量增长后最先暴露的问题并不只是存储容量，而是如何以合适方式组织查询与分析路径。

国内研究的第三个方向集中在异常检测、风险监测和预警解释。时间序列异常检测综述总结了点异常、区间异常和关联异常的主要类型及方法，并指出解释性、概念漂移和场景适配仍是核心难点。[9] 面向时序数据异常检测的可视分析综述则强调，多视图联动、上下文对照和交互过滤对于理解复杂异常具有重要意义。[7] 数字化转型背景下金融风险监测与预警体系研究以及中国股票市场操纵行为及预警机制研究，进一步将研究视角拉向金融风险表征、跨主体协同预警和市场操纵识别，说明风险识别已经由静态判断走向多因子、动态化和更强解释性的综合表达。[13][14]

总体来看，国内研究已经在平台建设、时序查询优化和风险预警应用方面取得了较充分进展，但这些成果往往分散在不同方向中，缺少围绕同一项目同时整合数据接入治理、多面板分析、高性能查询和风险雷达的统一实现范式。本文的系统设计正是在这一交叉位置上继续推进。

##### 1.2.2 国外研究方向及进展

国外研究在基础方法和底层支撑技术方面更具代表性。A general approach for running Python codes in OpenFOAM using an embedded pybind11 Python interpreter 从工程角度说明了 pybind11 在 C++ 系统中嵌入 Python 解释器和复用 Python 能力的可行性，这为本文采用“Python 业务 + C++ 核心算法”的协同模式提供了直接参考。[1] 对本文而言，这类研究的价值不在于其具体应用场景，而在于说明轻量级绑定机制足以支撑跨语言能力接入。

在顺序统计和近似分位方向，Computing Extremely Accurate Quantiles Using t-Digests 提出了面向分位查询的紧凑摘要结构，并强调其在尾部分位上的较高精度；数据流上的分位数近似算法研究则从更一般的数据流问题出发，说明近似分位结构在空间受限场景中的实用意义。[2][12] 这些研究为本文中的近似区间第 K 大查询提供了理论基础，也帮助明确了 t-digest 在本系统中的定位，即作为近似方案而非精确替代方案。

在精确顺序统计所依赖的结构基础方面，Making data structures persistent 系统阐释了数据结构持久化的基本思想，即通过保留旧版本和共享结构节点支持历史版本访问。[3] 这一思想对区间第 K 大这类依赖前缀版本差分的问题具有直接启发意义。与此同时，PISA: An Index for Aggregating Big Time Series Data 说明了面向大规模时间序列聚合查询时，专门设计索引结构具有显著价值。[4] 对本文来说，这些国外研究共同构成了高性能时序查询与顺序统计能力的底层方法支撑。

与国内研究相比，国外研究更偏向于给出通用方法、摘要结构和系统级索引思路，而较少直接讨论如何把这些能力落到一个带有用户认证、批次管理、分析页面和风险解释页面的完整业务系统中。也正因如此，本文的工作重点不是重复提出新算法，而是在已有理论基础上完成系统级整合。

##### 1.2.3 存在问题分类

结合国内外研究现状，可以将当前不足归纳为三类。第一类不足是平台能力与算法能力分离较明显。平台型研究更关注业务流程、页面展示和指标组织，算法型研究则更关注单一结构或单一计算问题，二者之间缺少一条清晰的工程落地路径。对于股票交易数据场景而言，如果系统只能展示数据而缺少高性能查询能力，或者只能展示算法结果却缺少批次管理与用户隔离，同样难以构成完整方案。

第二类不足是时序查询优化与风险分析表达之间衔接不足。已有研究已经分别讨论了时序数据库、异常查询优化、异常检测和风险预警，但较少在同一系统内同时解决批次化数据管理、多面板分析、精确/近似查询切换以及多维风险事件组织等问题。换言之，现有研究更像是“局部强、整体弱”，而不是从数据接入到风险解释的连续体系。

第三类不足是异常结果的解释能力仍然偏弱。许多研究能够识别异常，却没有进一步把异常事件组织为分级结果、标的画像、日期聚合和上下文钻取视图。在金融场景中，用户真正关心的不只是“有没有异常”，还包括“异常为什么发生、发生在什么范围、与历史相比处于什么位置、是否值得进一步关注”。本文的风险雷达设计就是针对这一不足展开的系统化尝试。

##### 1.2.4 发展趋势和研究展望

综合现有研究可以看出，相关领域正在沿着四个方向继续发展。第一，金融数据平台将继续向一体化演进，即把数据接入、治理、分析与预警放入同一系统中组织，而不是依赖多个彼此割裂的工具。[5][6][10][13] 第二，时序查询与异常检测会更加重视高性能与低延迟，围绕时序索引、近似分位和多维查询优化的研究仍将持续深化。[2][4][8][11][12]

第三，跨语言协同会成为工程实现中的重要趋势。随着 Python 在数据分析和 Web 开发中的广泛使用，以及 C++ 在性能敏感模块中的稳定优势，以 pybind11 为代表的轻量级桥接方案有望继续支撑“高性能模块 + 高开发效率业务系统”的组合。[1] 第四，风险识别将更加重视多因子联合表达、可视化解释和事件上下文，即从单一异常检测逐步走向异常排序、分级、画像与钻取的综合表达。[7][9][13][14]

因此，面向未来的研究重点不再只是单点算法改进，而是如何在真实系统中把时序数据管理、精确/近似查询、多维异常排序和风险解释进一步融合。本文所设计系统正是在这一趋势下展开的项目化实践。

#### 1.3 研究目标与主要内容

本文的目标是设计并实现一个面向股票历史交易数据的全栈系统，在统一工程框架下打通注册登录、文件导入、交易查询、多面板分析、算法增强和风险雷达展示。系统既要解决数据接入和管理问题，也要解决区间查询、近似顺序统计和异常解释问题。

围绕这一目标，本文的主要研究内容包括以下几个方面。首先，构建用户认证与权限隔离机制，使不同用户的数据访问边界清晰，并支持管理员在更高作用域内完成系统级查看。其次，建立以用户上传历史交易文件为核心的数据导入链路，实现 CSV/XLSX 文件接收、表头映射、字段校验、批次管理、软删除复用和原始产物保留，使后续分析能够依附于统一批次对象展开。

再次，围绕交易数据本身构建分析中心，实现摘要、质量、指标、风险、异常、横截面、相关性和范围对比等多类分析能力，使用户能够从基础统计、时序行为和跨范围差异三个角度理解数据。与此同时，本文通过 pybind11 将区间最大成交额、区间第 K 大成交量和联合异常排序三类 C++ 算法能力接入 Python 后端，其中区间第 K 大查询同时支持精确和近似两套方案。[1][2][3][12]

最后，本文以联合异常排序为基础，进一步构建三维风险雷达，实现风险事件总览、严重度分级、标的画像、日期聚合和事件上下文钻取，并通过后端测试、C++ 单元测试、Python 绑定 smoke test、前端构建验证以及 benchmark 脚本支撑，对系统正确性和实验边界进行分析。

#### 1.4 技术路线与研究方法

本文采用“需求分析驱动、分层架构实现、跨语言算法增强、测试验证闭环”的技术路线。前端使用 Vue 3、TypeScript、Element Plus 和 ECharts 组织页面与图表；后端使用 FastAPI、SQLAlchemy 2、Pydantic 2 和 Pandas 实现接口、建模与分析逻辑；数据库使用 PostgreSQL 保存用户、批次和交易记录；算法模块使用 C++ 与 pybind11 构建，并通过 `algo_bridge` 接入 Python 服务层。[1][8][11]

在研究方法上，本文综合运用文献研究法、系统设计法、案例实现法和实验验证法。文献研究用于支撑平台设计、时序查询、近似分位、持久化结构和异常解释等理论基础；系统设计法用于梳理模块边界、数据流转关系与接口分组；案例实现法强调全部论述都以当前仓库中的真实页面、接口、服务和测试为依据；实验验证法则用于说明系统已经证明了什么、当前还不能声称什么。[2][3][7][8][9][11][13]

#### 1.5 论文结构安排

全文共分七章。第 1 章介绍研究背景、研究现状、研究目标和技术路线；第 2 章介绍系统使用的技术与理论基础；第 3 章分析用户角色、功能需求和业务流程；第 4 章给出系统总体设计；第 5 章说明认证、导入、分析、算法和风险雷达等核心实现；第 6 章总结测试与实验分析；第 7 章给出全文总结、创新点、局限和后续展望。

### 第 2 章 相关技术与理论基础

#### 2.1 前端技术栈

##### 2.1.1 Vue 3 组合式 API 与组件化开发

系统前端采用 Vue 3 组合式 API 组织页面逻辑，以便将筛选条件、异步请求、图表配置和局部状态按功能维度拆分。对于分析中心和风险雷达这类多区域联动页面，组合式 API 能够较好地平衡复杂度和可维护性。

##### 2.1.2 TypeScript 类型安全与工程规范

TypeScript 用于约束接口响应、页面状态和组件属性，使批次统计、交易记录、分析结果和风险事件等多种数据结构能够在前端保持清晰边界。对于本系统而言，类型安全的价值主要体现在减少前后端联调过程中的字段错误。

##### 2.1.3 Vue Router 路由守卫与导航控制

前端通过 Vue Router 组织页面跳转，并借助全局路由守卫控制未登录状态下对业务页面的访问。这样一来，登录页、注册页与各业务页面之间形成清晰导航边界，与后端鉴权共同构成访问控制机制。

##### 2.1.4 Element Plus UI 组件库

Element Plus 提供表单、表格、分页、对话框和反馈组件，适合构建业务系统型页面。系统中的上传表单、统计卡片、筛选区域、记录列表和交互提示均建立在该组件库基础之上。

##### 2.1.5 ECharts 数据可视化引擎

ECharts 用于指标走势、相关性矩阵、异常散点图和日期聚合图等可视化输出。异常可视分析研究指出，多视图和上下文相关的图形表达对于理解时序异常十分重要，这也是本系统在分析中心和风险雷达中大量采用 ECharts 的原因。[7]

#### 2.2 后端技术栈

##### 2.2.1 FastAPI 异步框架与路由组织

FastAPI 用于组织认证、导入、交易查询、分析中心和算法相关接口。其优势在于类型注解友好、接口开发效率较高，适合当前以前后端分离为主的系统形态。

##### 2.2.2 SQLAlchemy 2 ORM 与 Mapped 声明式建模

SQLAlchemy 2 用于定义用户、批次、清单、产物和交易记录等核心实体，并通过 ORM 机制统一数据库访问方式。这种建模方式有助于保持数据库结构与业务对象的一致性。

##### 2.2.3 Pydantic 2 数据校验与序列化

Pydantic 2 负责请求参数校验和响应结构序列化，使登录、上传、筛选、算法调用和风险雷达结果等接口能够在边界上保持稳定。对于多模块系统而言，这一层能有效降低无效参数直接进入服务逻辑的风险。

##### 2.2.4 Alembic 数据库版本迁移

Alembic 用于维护数据库表结构变化，使项目在不同开发环境中保持模式一致。对于毕业设计项目而言，这一能力有助于减少环境差异导致的结构问题。

##### 2.2.5 Pandas 时间序列处理与指标计算

Pandas 是分析中心的主要计算工具，用于实现日期排序、窗口统计、技术指标、风险指标和异常基线等逻辑。它承担的是业务分析层计算，而不是替代数据库或算法模块。

#### 2.3 数据库技术

##### 2.3.1 PostgreSQL 关系型数据库

本文处理的是典型时序交易数据，但系统并未直接实现专用时序数据库，而是面向“用户上传—批次管理—记录查询—分析调用”的项目场景，采用 PostgreSQL 作为统一持久化后端。时序数据库综述和异常检测查询优化研究指出，时序数据管理需要同时关注写入、查询和延迟控制等问题。[8][11] 对本项目而言，关系型数据库更适合表达用户、批次、文件产物和交易记录之间的结构关系。

##### 2.3.2 条件唯一索引与软删除设计

在此基础上，系统进一步使用条件唯一索引和软删除机制表达业务语义。例如，活动状态下的同名数据集在同一用户作用域内不能重复，而删除后的名称可以复用；同一批次下同一标的同一日期的交易记录不能重复。这样既保证一致性，也便于后续追溯。

#### 2.4 身份认证与权限控制

##### 2.4.1 JWT Bearer Token 认证机制

系统采用 Bearer Token 作为无状态认证机制，在用户登录后签发带有身份信息和过期时间的令牌，并由后端依赖层统一解析和校验。这种方式适合前后端分离系统的持续请求访问。

##### 2.4.2 PBKDF2-SHA256 密码哈希

密码存储采用 PBKDF2-SHA256 方案，避免明文密码进入数据库，同时提升基本账户安全性。注册和登录阶段分别完成哈希生成与比对。

##### 2.4.3 用户/管理员双角色权限模型

权限模型区分普通用户与管理员：普通用户仅能访问自身批次及其分析结果，管理员可以在更高作用域内查看全站批次和统计信息，从而支撑系统验证与统一管理。

#### 2.5 算法理论基础

##### 2.5.1 线段树与区间最大值查询（RangeMaxSegmentTree）

线段树适合处理区间最值问题，能够在保留预处理结构的基础上快速回答给定区间的聚合查询。PISA 等时序索引研究说明，围绕区间聚合结果进行预组织是提升时序查询效率的重要思路。[4]

##### 2.5.2 可持久化线段树与区间第 K 大查询（RangeKthPersistentSegmentTree）

可持久化结构通过保留历史版本和共享节点，为区间第 K 大这类顺序统计问题提供了高效精确解法基础。[3] 本文中的精确顺序统计能力正建立在这一思想之上。

##### 2.5.3 t-digest 近似分位与近似顺序统计

t-digest 通过紧凑摘要结构近似表示数据分布，适合在响应优先场景下作为精确方案的补充。[2] 数据流上的分位数近似算法研究也支持其在空间受限场景中的应用价值。[12]

##### 2.5.4 CDQ 分治与二维/三维历史支配计数（HistoricalDominanceCDQ / 3D）

时间序列异常检测、异常可视分析和金融风险监测研究共同说明，多维异常识别比单指标规则更能贴近真实风险场景。[7][9][13][14] 本文据此在工程实现中使用二维和三维历史支配计数来完成联合异常排序与风险雷达事件分级。

#### 2.6 跨语言集成技术

##### 2.6.1 pybind11 C++/Python 绑定机制

相关研究表明，pybind11 可以较轻量地建立 C++ 与 Python 的接口桥梁，使原生 C++ 代码能够以 Python 模块形式被直接加载和调用。[1] 当前项目即通过这一机制导出 `algo_module_py`，再由 `backend/app/algo_bridge/` 完成输入转换与结果适配。

##### 2.6.2 CMake 构建系统与 CTest 测试框架

算法模块采用独立的 `algo-module` 目录组织，通过 CMake 管理构建，并使用 CTest 执行 C++ 单元测试和 Python 绑定 smoke test。这样的组织方式既保证了底层实现独立可维护，也保证了跨语言链路可验证。

### 第 3 章 系统需求分析

#### 3.1 用户角色与业务场景

##### 3.1.1 普通用户：注册、登录、上传交易文件、查看分析结果

普通用户负责注册、登录、上传交易文件、查看导入批次、浏览记录并使用分析中心和风险雷达。对该角色而言，系统最重要的要求是数据围绕个人批次展开，且操作路径清晰、反馈明确。

##### 3.1.2 管理员：全站批次可见、跨用户数据管理与验证

管理员在普通用户能力基础上具有全站批次可见能力，用于系统验证和统一管理。管理员仍通过同一套业务入口访问系统，只是在数据作用域上高于普通用户。

#### 3.2 功能需求分析

##### 3.2.1 用户鉴权需求（注册、登录、当前用户信息）

系统需要支持注册、登录和当前用户信息恢复，以保证业务访问建立在有效身份之上。前端刷新后仍应恢复登录态，后端则应统一校验令牌合法性。

##### 3.2.2 数据导入管理需求（CSV/XLSX 上传、数据集命名、导入历史、统计、软删除）

系统需要支持 CSV/XLSX 上传、表头映射、字段校验、批次创建、导入历史查看和软删除复用。导入不仅是文件接收，还包括生成可被后续查询和分析复用的批次对象。

##### 3.2.3 交易记录查询需求（标的列表、记录列表、分页）

系统需要支持按批次获取标的列表，并按标的和日期范围查询分页记录，为分析和算法调用提供基础数据，同时保证页面在记录量增加后仍具备可用性。

##### 3.2.4 分析中心需求（八大分析面板）

系统需要围绕同一筛选范围提供摘要、质量、指标、风险、异常、横截面、相关性和范围对比八类结果，并保证这些结果在同一页面中保持口径一致。

##### 3.2.5 C++ 算法增强需求

系统需要支持区间最大成交额、区间第 K 大成交量和联合异常排序，并允许精确/近似方案切换。这样才能体现高性能模块在真实业务链路中的价值。

##### 3.2.6 风险雷达需求

系统需要支持索引状态查看、重建、事件总览、事件过滤、标的画像、日期聚合和事件上下文钻取，从而把异常检测结果提升为可解释、可浏览的风险视图。

#### 3.3 非功能需求分析

##### 3.3.1 正确性：导入校验、字段合法性、权限隔离、算法精度

导入校验、权限隔离和算法结果都应可靠，尤其要避免脏数据进入主链路，并保证精确算法结果与测试预期一致。

##### 3.3.2 性能：C++ 算法查询效率、索引缓存与快照复用

关键查询不能退化为全量扫描，风险雷达需要通过索引、缓存和快照复用降低重复构建成本。

##### 3.3.3 可维护性：前后端分层、路由与服务分离

系统需要在前端、后端和算法模块之间保持清晰边界，确保功能增加后仍能稳定维护。

##### 3.3.4 可扩展性：算法模块可通过 pybind11 持续扩展

未来若新增更多区间统计或异常排序能力，应能够通过 pybind11 和 `algo_bridge` 增量接入，而不需要重构整套系统。

##### 3.3.5 可用性：交互友好、错误提示明确、可视化结果直观

页面交互需要清晰、错误提示明确、结果展示直观，尤其是分析中心和风险雷达应便于用户连续使用。

##### 3.3.6 安全性：密码哈希、Token 鉴权、数据按用户隔离

系统必须保证密码不以明文存储、令牌能够被统一校验、用户数据不会因越权访问而泄露。

#### 3.4 业务流程与用例分析

系统可以抽象为四条主链路：认证链路、导入链路、分析链路和风险雷达链路。用户先注册登录，再上传交易文件形成导入批次；随后在交易管理页浏览标的和记录，进入分析中心查看多维结果，必要时再通过风险雷达查看异常事件总览、画像和上下文。该流程说明本系统的核心价值在于形成完整闭环，而不是提供彼此分散的独立功能。

### 第 4 章 系统总体设计

#### 4.1 系统总体架构

系统采用前端展示层、后端服务层、数据持久化层和 C++ 算法模块层四层架构。前端负责页面交互和图表展示；后端负责鉴权、业务编排和分析服务；数据库负责持久化用户、批次和交易记录；算法模块负责区间统计和异常排序等性能敏感任务。用户请求经前端页面发起后，由 FastAPI 路由进入服务层，随后根据场景访问 PostgreSQL、执行 Pandas 分析或通过 `algo_bridge` 调用 `algo_module_py`，最后再回到前端展示。

#### 4.2 前端模块设计

##### 4.2.1 页面模块划分

前端包含登录页、注册页、系统总览页、交易数据管理页、分析中心页和风险雷达页六类真实页面。登录页与注册页负责身份建立，总览页负责系统状态和导入概览，交易管理页负责上传、批次管理和算法卡片，分析中心页负责多面板分析，风险雷达页负责异常事件的可视化与钻取。

##### 4.2.2 通用组件设计（AppShell、StatCard、PanelCard、EChartPanel、EmptyState）

系统抽象了布局容器、统计卡片、面板容器、图表容器和空状态等通用组件，以统一页面风格和交互结构。这些组件让分析中心和风险雷达中的多块面板能够保持一致布局。

##### 4.2.3 前端 API 封装层设计（auth / imports / trading / analysis / riskRadar / health）

前端通过分业务域的 API 模块统一处理请求，而不是在页面中直接拼接接口地址。这使认证、导入、交易查询、分析和风险雷达都能够围绕稳定的数据访问边界组织。

##### 4.2.4 状态管理设计（auth store / runtime store）

认证状态用于保存用户身份和令牌，运行时状态则围绕当前批次、筛选条件和局部页面行为展开。二者分离使状态管理更清晰。

##### 4.2.5 路由守卫与登录态持久化设计

前端通过路由守卫限制未登录状态下对业务页面的访问，并通过本地持久化恢复刷新后的会话状态，从而提高使用连贯性。

#### 4.3 后端模块设计

##### 4.3.1 路由层设计（api/routes）

后端路由按业务域分组，包括认证、健康检查、导入管理、交易查询、分析中心以及算法相关子路由。路由层负责接收请求、校验参数并调用服务层。

##### 4.3.2 服务层设计（services）

服务层对应 `auth.py`、`imports.py`、`trading_analysis.py`、`algo_trading.py`、`algo_indexes.py` 和 `risk_radar.py` 等核心文件，负责承接认证、导入、分析、算法和风险雷达逻辑。

##### 4.3.3 数据访问层设计（repositories）

当前项目的数据访问并未做成复杂仓储体系，但设计上仍遵循按实体职责组织查询与写入逻辑的原则，以保持后端结构清晰。

##### 4.3.4 数据模型层设计（schemas / models）

模型层由 ORM 实体和 Pydantic 模型组成。前者负责数据库结构表达，后者负责接口输入输出契约，从而实现“存储结构”和“通信结构”的分离。

##### 4.3.5 核心基础层设计（core：config / database / security）

配置管理、数据库连接和安全能力被统一下沉到基础层，避免业务服务重复处理公共逻辑。

#### 4.4 数据库设计

##### 4.4.1 E-R 图

虽然本文不绘制图形化 E-R 图，但系统关系清晰：`users` 与 `import_runs` 为一对多，`import_runs` 与 `import_manifests`、`import_artifacts` 及 `trading_records` 也为一对多。

##### 4.4.2 users 表设计

`users` 表保存用户名、密码哈希、角色及时间戳，是系统认证和权限控制的基础。

##### 4.4.3 import_runs 表设计

`import_runs` 是最核心的业务主表，用于表达一次完整导入批次，记录所属用户、数据集名称、导入状态、来源和软删除标记等信息。

##### 4.4.4 import_manifests 表设计

`import_manifests` 保存导入过程中的结构化清单信息，用于保留导入上下文。

##### 4.4.5 import_artifacts 表设计

`import_artifacts` 将磁盘中的上传文件与数据库中的导入批次关联起来，记录文件名、路径和媒体类型等元数据。

##### 4.4.6 trading_records 表设计

`trading_records` 保存实际交易数据，包括标的代码、交易日期、开高低收、成交量和成交额等字段，是查询、分析和风险识别的底层数据来源。

##### 4.4.7 索引与约束策略

系统使用条件唯一索引和复合唯一约束保证业务一致性，例如活动状态下同一用户不能拥有同名数据集，同一批次下同一标的同一日期的记录不能重复。

#### 4.5 核心接口设计

##### 4.5.1 鉴权接口组（register / login / me）

鉴权接口组用于完成注册、登录和当前用户信息恢复，是后续业务访问的入口。

##### 4.5.2 导入管理接口组（runs / stats / trading / 软删除）

导入管理接口组负责文件上传、批次列表、统计信息和软删除操作，是系统主业务入口。

##### 4.5.3 交易查询接口组（instruments / records）

交易查询接口组负责按批次获取标的列表，并按标的和日期范围获取交易记录。

##### 4.5.4 分析中心接口组（summary / quality / indicators / risk / anomalies / cross-section / correlation / compare-scopes）

分析中心接口组负责八类分析结果输出，是分析页面联动展示的主要数据来源。

##### 4.5.5 算法接口组（range-max-amount / range-kth-volume / joint-anomaly-ranking）

算法接口组负责区间最大值、第 K 大和联合异常排序能力，使高性能模块以独立能力对外暴露。

##### 4.5.6 风险雷达接口组（索引状态 / 重建 / 总览 / 事件列表 / 标的列表 / 日期聚合 / 事件上下文）

风险雷达接口组围绕索引准备和结果消费两条逻辑线展开，支撑风险雷达页面完整运行。

#### 4.6 C++ 算法模块与 Python Bridge 集成设计

##### 4.6.1 algo-module 目录组织与构建方式

当前项目将 C++ 实现集中在 `algo-module` 目录下，通过独立的源码、绑定代码和测试结构保持模块边界清晰。

##### 4.6.2 pybind11 绑定与 algo_module_py 模块导出

系统通过 pybind11 导出 `algo_module_py`，使 C++ 数据结构能够以 Python 模块形式被后端直接加载。[1]

##### 4.6.3 algo_bridge 适配层设计

`backend/app/algo_bridge/` 负责输入转换与结果适配，把底层算法能力稳定接入业务系统。

#### 4.7 风险雷达索引与缓存设计

##### 4.7.1 索引生命周期状态机（pending → building → ready / failed）

风险雷达通过 `pending / building / ready / failed` 四态管理索引构建过程，使页面能够感知索引准备状态。

##### 4.7.2 InstrumentAlgoIndex 对象结构（含 RangeMax/RangeKth 线段树、t-digest 分块索引）

系统为单个标的构建 `InstrumentAlgoIndex`，其中包含区间最大值结构、顺序统计结构和 t-digest 摘要，为后续风险分析提供共享底座。

##### 4.7.3 RiskRadarIndexCache 对象结构（概览、事件、标的画像、日历、标的索引）

`RiskRadarIndexCache` 不仅保存原始索引，还保存已组织好的总览、事件、画像和日历结果，以减少重复计算。

##### 4.7.4 快照写入与加载复用机制

系统通过快照与内存缓存共同降低风险雷达的重复构建开销，使页面多次进入时能够快速恢复结果。

##### 4.7.5 三维 CDQ 支配计数在事件分级中的应用

系统将收益率标准分数、成交量倍数和振幅倍数组成三维特征，并据此计算事件的历史相对异常程度，用于严重度分级。[13][14]

##### 4.7.6 事件上下文解释生成（窗口分位、分布变化、局部成交额峰值）

异常可视分析研究强调，异常点只有置于上下文中才更容易被理解。[7] 因此，系统在索引层预留了窗口分位、分布变化和局部峰值查询能力。

### 第 5 章 系统详细实现

#### 5.1 用户认证与权限隔离实现

##### 5.1.1 注册与登录接口实现

注册接口负责用户名校验、规范化和密码哈希写入；登录接口负责密码校验并返回令牌与当前用户信息。二者共同构成系统的认证入口。

##### 5.1.2 JWT 签发与校验流程

用户登录后，后端签发访问令牌，后续请求由依赖层统一解析和校验。这样既保持前后端分离，也使业务接口只关注实际逻辑。

##### 5.1.3 PBKDF2-SHA256 密码哈希实现

密码不以明文形式存储，而是在注册时完成哈希生成，在登录时完成哈希比对，保证基本安全性。

##### 5.1.4 普通用户/管理员访问控制依赖注入

后端在依赖层区分普通用户与管理员作用域，使数据访问边界不依赖前端按钮控制，而由服务端统一保障。

##### 5.1.5 前端路由守卫与 Token 持久化

前端通过 `auth` store 持久化保存令牌和用户资料，并通过路由守卫限制未登录状态下对业务页面的访问。

#### 5.2 文件上传与数据导入实现

##### 5.2.1 上传文件保存路径组织（data/uploads/trading/{user_id}/{run_id}/）

用户上传的 CSV 或 XLSX 文件会被保存到 `data/uploads/trading/{user_id}/{run_id}/` 路径下，使文件系统对象与数据库批次形成对应关系。

##### 5.2.2 中英文别名表头标准化映射（trading_head.json）

服务层根据 `trading_head.json` 完成中英文别名表头映射，从而兼容不同来源的交易文件模板。

##### 5.2.3 必填列、可选列与重复记录校验

系统会对必填列、可选列、日期格式、数值合法性和重复记录进行校验，避免脏数据直接进入正式表。

##### 5.2.4 导入批次、manifest、artifact 与交易记录写入流程

通过校验后，系统依次写入 `import_runs`、`import_manifests`、`import_artifacts` 和 `trading_records`，保留完整导入上下文。

##### 5.2.5 导入失败标记、软删除与数据集名称复用机制

导入失败批次会被标记并隐藏，删除操作采用软删除，从而允许后续复用原数据集名称。

##### 5.2.6 命令行导入脚本实现（import_data.py）

项目还提供 `backend/scripts/import_data.py` 作为辅助导入工具，说明导入能力已沉淀为可复用服务而非单一页面逻辑。

#### 5.3 交易记录查询与统计总览实现

##### 5.3.1 批次列表查询与统计信息生成

后端根据用户角色返回可见批次及其统计信息，为总览页和交易管理页提供基础数据。

##### 5.3.2 标的列表与交易记录查询

系统支持按批次获取标的列表，并按标的和日期范围分页查询记录，为分析和算法调用提供统一数据入口。

##### 5.3.3 总览页面：健康状态、按月导入趋势、最近导入记录

`OverviewPage.vue` 将健康状态、统计卡片、按月导入趋势和最近导入记录集中展示，使用户在进入分析页前先获得系统级概览。

#### 5.4 交易分析中心实现

##### 5.4.1 交易摘要分析（统计最高价、最低价、均价、总成交量等）

摘要分析返回最高价、最低价、平均收盘价、总成交量和总成交额等基础统计，用于勾勒所选范围内的交易轮廓。

##### 5.4.2 数据质量分析（覆盖率、缺失日期、OHLC 合法性、零值检测、扁平日）

质量分析评估覆盖率、缺失日期、OHLC 逻辑和异常零值，并识别开高低收完全相同的扁平交易日，以判断数据是否适合后续分析。

##### 5.4.3 技术指标分析（MA5/10/20、EMA12/26、MACD/Signal/Histogram、RSI14、Bollinger Bands、ATR14）

系统依托 Pandas 生成 MA、EMA、MACD、RSI、Bollinger Bands 和 ATR 等技术指标，为趋势分析和波动观察提供基础。

##### 5.4.4 风险指标分析（区间收益率、波动率、最大回撤及持续天数、上涨日占比、最大单日涨跌幅）

风险分析围绕收益和波动展开，重点输出收益率、波动率、最大回撤及其持续时间、上涨日占比和最大单日涨跌幅等结果。

##### 5.4.5 异常检测分析（放量异常 2× 基线、收益率异常 3σ、振幅异常 2× 基线、突破前高/前低）

系统使用滚动基线和阈值规则识别放量、收益率、振幅和突破事件，使异常结果具备较好的可解释性，并能与后续风险雷达顺利衔接。[7][9]

##### 5.4.6 横截面排序分析（按 total_return / volatility / total_volume / total_amount / average_amplitude 多指标排序）

横截面排序用于比较同一批次内不同标的在收益、波动和活跃度方面的差异，是从纵向时序分析扩展到横向比较的重要补充。

##### 5.4.7 收益率相关性矩阵分析

相关性分析通过对齐标的间重叠日期并计算收益率相关系数，帮助用户识别标的之间的联动关系。

##### 5.4.8 范围对比分析（标的重叠、记录重叠、逐字段不一致检测与样例展示）

范围对比用于比较不同批次或不同筛选范围下的数据差异，帮助用户理解两组结果不一致的来源。

#### 5.5 C++ 算法查询实现

##### 5.5.1 区间最大成交额实现

区间最大成交额查询由 `algo-module` 中的 `RangeMaxSegmentTree` 完成，前端通过交易管理页触发请求，后端通过 `algo_bridge` 完成数据缩放与索引映射，再返回最大成交额及其命中信息。[1][4]

##### 5.5.2 区间第 K 大成交量实现

区间第 K 大查询分为两套方案。精确方案依赖 `RangeKthPersistentSegmentTree`，通过持久化结构和前缀版本差分恢复区间值频信息。[3] 近似方案依赖 `RangeKthTDigestBlockIndex`，通过摘要结构估计目标顺序统计值，并以 `is_approx` 和说明字段明确其近似性质。[2][12] 两套方案通过统一接口和 `method` 参数切换。

##### 5.5.3 联合异常排序实现

联合异常排序首先在 Python 侧生成 `return_z20 × volume_ratio20` 二维事件序列，再通过二维历史支配计数计算 `joint_percentile` 与严重度等级，并按 Top-N 输出结果。该设计使联合异常不再是单点规则判断，而成为可比较的排序结果。[7][9]

#### 5.6 三维风险雷达实现

##### 5.6.1 索引构建实现

风险雷达首先按标的加载序列，构建成交额区间最大值结构、成交量/振幅顺序统计结构和 t-digest 摘要，并同步计算 `return_z20`、`volume_ratio20` 和 `amplitude_ratio20` 三维特征，为后续风险组织提供统一底座。

##### 5.6.2 缓存与快照实现

系统通过 `AlgoIndexManager` 管理索引生命周期，并结合内存缓存、磁盘快照和 `reuse_count` 统计降低重复构建开销。这使风险雷达更接近预构建分析视图，而不是每次都从零计算的临时查询。

##### 5.6.3 总览与事件查询实现

`risk_radar.py` 在缓存结果基础上组织风险总览、事件列表、标的画像和日期聚合结果，使页面能够从整体风险水平、关键事件、重点标的和风险日期四个角度浏览数据。

##### 5.6.4 事件上下文钻取实现

在单个事件钻取时，系统会计算多窗口分位、前后分布变化、局部成交额峰值，并生成 `cause_label`。这样风险雷达不只告诉用户“哪里异常”，还进一步说明“异常为什么值得关注”。[6][7][13][14]

#### 5.7 前端页面实现

##### 5.7.1 登录页与注册页实现

登录页和注册页构成认证入口，分别负责用户身份建立、错误提示和页面跳转。

##### 5.7.2 系统总览页实现（统计卡片、健康状态、月度趋势、最近导入）

系统总览页通过卡片和趋势图展示健康状态、批次规模和最近导入情况，是用户进入系统后的第一观察窗口。

##### 5.7.3 交易数据管理页实现（文件上传、批次管理、区间算法卡片、算法切换）

交易数据管理页承担上传、批次切换、记录浏览和区间算法调用四类功能，是数据接入与高性能查询的结合点。

##### 5.7.4 分析中心页实现（六大区域联动：摘要/指标/风险/异常/横截面相关性/质量对比）

分析中心页以统一筛选条驱动多个面板联动刷新，使摘要、质量、指标、风险、异常和对比结果在同一范围内保持一致。

##### 5.7.5 风险雷达页实现（异常散点图、事件榜单、标的画像、日历热力图、事件钻取面板）

风险雷达页负责异常散点图、榜单、画像、日期聚合和事件上下文的统一展示，是系统最复杂也最具特色的前端页面。

### 第 7 章 总结与展望

#### 7.1 工作总结

本文围绕股票交易数据管理与分析系统完成了从需求分析、总体设计到详细实现和测试验证的完整工作。系统以用户上传历史交易文件为入口，构建了注册登录、权限隔离、文件导入、批次管理、交易查询、分析中心、算法查询和风险雷达等模块，形成了一个可运行的全栈业务闭环。

#### 7.2 创新点与贡献

本文围绕三维风险特征构建风险雷达，将联合异常排序扩展为事件分级、标的画像、日期聚合和上下文钻取体系。[6][7][13][14]
