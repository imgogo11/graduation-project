# 毕业论文目录与写作提纲

## 文档用途

本提纲用于将当前仓库已经落地的项目主线，直接映射为毕业论文可用的目录结构与写作清单。默认论文题目采用：

**《基于 C++ 算法模块的股票交易数据管理与分析系统设计与实现》**

本文档遵循以下默认取舍：

- 论文类型定位为“软件系统设计与实现”，算法模块作为核心亮点和创新点。
- 全文只围绕当前真实主链路展开，不再书写早期已退场的电商、爬虫、synthetic 或历史演示分支。
- 章节采用本科工科常见的 7 章结构，不主动压缩成 5 章。
- 第 4 章强调“设计”，第 5 章强调“实现”，第 6 章同时承担“系统测试 + 算法实验”。

## 写作边界

论文写作时建议明确以下边界，避免范围失控：

- 研究对象是股票历史交易数据，不扩展到电商、舆情、多模态等其他领域。
- 数据来源以用户上传 `CSV` / `XLSX` 文件为主，不将外部实时抓取链路写成系统主功能。
- 系统核心目标是“数据管理 + 交易分析 + 算法增强”，不是通用量化交易平台。
- 风险雷达与区间算法为系统增强模块，应写成“面向交易分析场景的算法能力落地”。
- `t_digest` 在本文中应表述为近似分位 / 近似第 K 大查询方案，不应表述为严格精确解。

### 第 1 章 绪论

#### 1.1 研究背景与意义

建议围绕以下角度展开：

- 股票历史交易数据规模持续增长，人工整理和简单表格分析效率低。
- 单纯的数据管理系统难以满足技术指标、风险识别、异常发现等分析需求。
- Python 适合业务开发与数据处理，但在部分区间查询、排序统计、联合异常计算场景下，C++ 更适合承担高性能算法模块。
- 将“业务系统 + 数据分析 + C++ 算法模块”统一到一个可运行系统中，具有教学和工程实践价值。

#### 1.2 国内外研究现状

可从四条线索组织：

- 金融交易数据管理系统研究现状。
- 股票分析与可视化平台研究现状。
- 高性能区间查询与顺序统计算法研究现状。
- 异常检测、联合异常排序与风险预警研究现状。

写作建议：

- 研究现状以“已有成果做了什么、仍存在哪些空缺”为主，不要写成技术名词堆砌。
- 结尾应自然引出本系统的研究定位：在可用系统中集成算法模块，并提供可视化分析与验证。

#### 1.3 研究目标与主要内容

本节可直接概括为以下几个目标：

- 构建一个支持注册登录、权限隔离和交易文件导入的数据管理系统。
- 为管理员提供普通用户账号管理能力，包括编辑、启用/禁用与受限删除。
- 实现交易记录查询、指标分析、风险分析、异常检测、横截面与相关性分析。
- 设计并集成 C++ 算法模块，支持区间最大成交额、区间第 K 大成交量和联合异常排序。
- 构建风险雷达索引与可视化页面，实现异常事件总览、榜单、股票画像和事件钻取。
- 通过测试与实验验证系统正确性、可用性和算法效果。

#### 1.4 技术路线与研究方法

建议用一张技术路线图配合说明：

- 前端采用 Vue 3 + TypeScript + Element Plus + ECharts。
- 后端采用 FastAPI + SQLAlchemy + Alembic + Pandas。
- 数据库采用 PostgreSQL。
- 算法模块采用 C++23 + pybind11，通过 Python Bridge 接入后端。
- 风险雷达采用索引构建、快照缓存、查询复用的实现方式。

#### 1.5 论文结构安排

按章节简要说明每章内容，用 1 段或 1 组短句概括即可。

### 第 3 章 系统需求分析

#### 3.1 用户角色与业务场景

建议固定为两类角色：

- 普通用户：注册、登录、上传交易文件、查看自己的导入历史和分析结果。
- 管理员：查看全站可见批次、按用户过滤、执行跨用户数据管理和验证。

可描述的典型业务场景：

- 用户上传股票历史交易文件并形成导入批次。
- 用户按批次和股票查看交易记录与指标分析。
- 用户调用算法接口得到区间查询和异常排序结果。
- 用户在风险雷达页面查看异常事件分布和事件上下文。

#### 3.2 功能需求分析

本节建议按模块展开：

- 用户鉴权需求：注册、登录、当前用户信息获取。
- 管理员用户管理需求：普通用户列表、账号编辑、密码重置、启用/禁用、无业务数据用户删除。
- 导入管理需求：CSV/XLSX 上传、数据集命名、导入历史、统计、软删除。
- 交易查询需求：股票列表、交易记录列表。
- 分析中心需求：摘要、质量、指标、风险、异常、横截面、相关性、范围对比。
- 算法增强需求：区间最大成交额、区间第 K 大成交量、联合异常排序。
- 风险雷达需求：索引状态、重建、异常榜单、股票画像、日期聚合、事件钻取。

#### 3.3 非功能需求分析

建议至少覆盖：

- 正确性：导入校验、字段合法性校验、权限隔离。
- 性能：算法查询效率、风险雷达索引复用能力。
- 可维护性：前后端分层、路由与服务模块分离。
- 可扩展性：算法模块可继续扩展新查询能力。
- 可用性：页面交互清晰、错误提示明确、可视化结果直观。
- 安全性：密码哈希、Token 鉴权、按用户数据隔离。

### 第 4 章 系统总体设计

#### 4.1 系统总体架构

建议采用四层结构表述：

- 表现层：Vue 前端页面与可视化组件。

- 数据层：PostgreSQL、ORM 模型、迁移脚本。
- 算法层：C++ 算法引擎、Python Bridge、风险雷达索引缓存。

#### 4.2 前后端分层与模块划分

建议分别说明：

- 前端模块：`Overview`、`Analysis Center`、`Algo Radar` 三个业务页面，以及通用组件、共享数据集上下文、管理员扩展区块和状态管理。
- 后端模块：`api/routes`、`services`、`repositories`、`schemas`、`models`。
- 算法集成模块：`algo_bridge`、`algo-module`、索引管理与风险雷达服务。

建议每张表都写：

#### 4.4 核心接口设计

建议按接口组说明：

- 鉴权接口：`/api/auth/register`、`/api/auth/login`、`/api/auth/me`

### 第 7 章 总结与展望

#### 7.1 工作总结

总结系统完成情况：

- 实现了完整的数据管理与分析主链路。
- 实现了前后端联动与可视化展示。
- 实现了 C++ 算法模块接入与风险雷达增强能力。

#### 7.2 创新点与不足

创新点可从以下角度提炼：

- 将 C++ 算法模块与 Python Web 后端集成。

不足建议实事求是：

- 部署方案仍偏开发环境。
- 实验规模和数据范围仍可继续扩展。
- 算法种类和页面性能还可继续优化。

#### 7.3 后续改进方向

建议从以下方向写展望：

- 增加更多算法能力和更系统化性能评测。
- 完善生产部署与监控。
- 扩展更丰富的数据源与分析维度。
