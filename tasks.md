# 毕业设计任务规划

## 1. 项目定位与题目建议

### 1.1 项目定位
本项目最终定位为一个面向**股票数据**与**电商数据**的多源数据管理与高性能分析平台，核心目标不是做一个普通的 CRUD 后台，而是构建一个同时具备以下能力的完整毕设系统：

- 多源数据采集、清洗、入库与统一管理
- 业务查询、统计分析与图表可视化
- 基于高级数据结构的高性能分析能力
- Python 与 C++ 协同的工程化实现能力

### 1.2 推荐题目
- 中文题目：基于 C++ / Python 的多源数据管理与高性能分析平台设计与实现
- 备选题目：面向股票与电商数据的算法增强型管理分析系统设计与实现
- 英文题目：Design and Implementation of a High-Performance Multi-Source Data Management and Analysis Platform Based on C++ and Python

### 1.3 建设目标
- 保留“股票 + 电商”双业务域，体现多源数据平台特征
- 采用前后端分离架构，形成完整的系统工程闭环
- 使用 PostgreSQL 作为主数据库，Redis 作为缓存与任务状态层
- 使用 C++ 实现核心算法引擎，使用 Python 承担业务与数据处理职责
- 通过算法实验与性能对比形成论文和答辩亮点

## 2. 选题价值与工作量说明

本课题的工作量和研究价值主要体现在五个方面：

- 数据域完整：同时覆盖股票时序数据和电商业务数据，包含导入、清洗、管理、查询、分析等全流程
- 系统层完整：包含前端、后端、数据库、缓存、算法引擎、部署方案与文档体系
- 算法层突出：线段树、主席树、分块、CDQ 分治都与真实业务场景绑定，不是单纯展示算法名称
- 工程化充分：涉及跨语言集成、任务调度、日志审计、目录规范、容器化部署
- 实验验证明确：需要完成正确性验证、性能对比、实验报告与论文支撑材料

因此，该题目既满足毕业设计对“可运行系统”的要求，也能体现较强的算法特色和研究深度。

## 3. 最终技术选型

### 3.1 技术框架

| 层次 | 技术选型 | 作用 |
| --- | --- | --- |
| 前端 | Vue3 + TypeScript + Element Plus + ECharts | 管理后台、图表展示、实验看板 |
| 后端 | FastAPI + SQLAlchemy + Pydantic | 业务 API、鉴权、数据服务 |
| 数据处理 | Python + Pandas | 数据清洗、转换、导入 |
| 算法引擎 | C++17/20 + PyBind11 | 高性能区间查询与离线分析 |
| 数据库 | PostgreSQL 15+ | 主业务数据持久化 |
| 缓存/任务状态 | Redis | 缓存、异步任务状态、队列支撑 |
| 调度 | Celery 或 APScheduler | 定时同步、导入任务、实验任务 |
| 部署 | Docker Compose | 本地开发与演示环境统一 |

### 3.2 默认约束
- 主数据库正式定为 `PostgreSQL 15+`
- 本机旧版 MySQL 不再作为主方案，只作为备选历史对比背景
- `PyBind11` 作为 C++ 与 Python 的主集成方案
- `gRPC` 仅作为扩展或性能对比项，不进入最小可交付主链路
- `Redis` 默认保留，因为它同时承担缓存、任务状态和异步支撑职责

### 3.3 语言职责划分
- `Python`：业务 API、权限控制、数据导入、ETL、调度、报表导出
- `C++`：区间查询、历史版本查询、分块统计、离线多维分析等核心算法能力
- `TypeScript`：界面交互、图表可视化、任务状态展示、实验结果呈现

## 4. 总体架构设计

### 4.1 总体架构

```text
前端展示层（Vue3 + TypeScript + ECharts）
        ↓
后端业务层（FastAPI）
        ↓
┌───────────────────────────────┬───────────────────────────────┐
│ 数据层（PostgreSQL + Redis）   │ 算法层（C++ + PyBind11）        │
│ 持久化、条件查询、日志、缓存      │ 区间统计、历史版本、离线分析      │
└───────────────────────────────┴───────────────────────────────┘
```

### 4.2 协同原则
- 普通业务查询优先走数据库
- 高频分析查询优先走算法引擎
- 后端负责统一调度，不让前端直接感知底层差异
- 数据库保存原始业务数据，算法引擎基于数据库数据构建内存索引

### 4.3 关键数据流
- 数据导入流：数据源 -> Python 清洗 -> PostgreSQL 入库 -> 算法索引构建/更新
- 普通查询流：前端 -> FastAPI -> PostgreSQL/Redis -> 前端
- 分析查询流：前端 -> FastAPI -> C++ 算法引擎 -> FastAPI 封装结果 -> 前端
- 实验评测流：前端/后端发起任务 -> 数据库查询方案与算法方案分别执行 -> 结果统一落库与展示

## 6. 算法与业务映射

### 6.1 线段树
- 对应场景：股票价格区间最值、成交量区间统计、电商销售额区间聚合
- 核心价值：解决高频区间查询问题
- 适合展示内容：区间最大值、最小值、总和、均值、批量更新

### 6.2 主席树
- 对应场景：历史版本快照、区间第 K 值、时间版本对比
- 核心价值：把“历史数据查询”从普通时间过滤提升为“版本化查询”
- 适合展示内容：某时间版本下的价格分布、第 K 大/小值、跨版本对比

### 6.3 分块
- 对应场景：高频局部更新、价格区间统计、销量区间统计
- 核心价值：在实现难度与性能之间做工程平衡
- 适合展示内容：中等复杂度区间统计与局部修改

### 6.4 CDQ 分治
- 对应场景：多因子选股、多条件离线筛选、批量分析任务
- 核心价值：体现高阶离线算法能力和多维偏序建模能力
- 适合展示内容：时间、价格、销量、行业等多维条件联合分析

## 8. 数据库概要设计

### 8.1 选型结论
数据库正式采用 `PostgreSQL 15+`，理由如下：

- 更适合分析型与扩展型项目
- 对 JSON、索引、分区、复杂查询支持更好
- 与 Docker Compose 配合更方便
- 适合后续实验扩展与论文描述

### 8.2 数据库职责
- 保存用户与权限数据
- 保存股票业务数据
- 保存电商业务数据
- 保存算法任务、日志、实验结果
- 为常规筛选、分页、排序和详情展示提供支撑

### 8.3 核心数据类别
- 用户与权限类
- 股票数据类
- 电商数据类
- 算法任务与审计类

### 8.4 索引设计思路
- 围绕时间、对象标识和高频筛选条件建立联合索引
- 围绕日志与实验记录建立查询索引
- 在必要时引入按时间分区或冷热分层思路

### 8.5 与算法层的关系
- 数据库存储原始数据与业务真值
- 算法层不直接替代数据库
- 算法层基于数据库数据构建索引结构，为分析型查询加速

## 9. 接口边界设计

### 9.1 前端到后端接口族
- `auth`：用户认证与权限
- `import`：数据导入与预处理任务
- `stock`：股票查询与分析
- `commerce`：电商查询与分析
- `algo`：算法增强型查询
- `benchmark`：性能对比实验

### 9.2 后端到 C++ 引擎能力边界
- `range_query`：区间统计类能力
- `version_query`：历史版本与第 K 查询类能力
- `block_stats`：分块统计与局部更新类能力
- `offline_multi_factor_query`：离线多维分析类能力

### 9.3 接口设计原则
- 前端不直接区分数据库查询和算法查询
- 后端统一进行参数校验、权限校验和结果封装
- 算法层只暴露稳定能力边界，不暴露内部实现细节
- 性能实验接口与业务接口分离，避免互相污染

## 13. 风险与规避

| 风险 | 规避方案 |
| --- | --- |
| 数据源不稳定 | 准备离线样例数据和模拟数据生成方案 |
| PostgreSQL 环境安装耗时 | 优先使用 Docker Compose 统一环境 |
| PyBind11 集成复杂 | 先打通最小桥接链路，再逐步扩展算法模块 |
| 前后端联调周期偏长 | 先固定接口族和响应结构，再并行开发 |
| 算法与业务脱节 | 所有算法都必须绑定明确业务场景 |
| 后期论文时间不足 | 从开发中期开始同步沉淀图表、截图和实验记录 |
